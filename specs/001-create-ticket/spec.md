# 기능 명세: 티켓 생성

**기능 브랜치**: `001-create-ticket`

**작성일**: 2026-07-24

**상태**: 초안

**입력**: 사용자 설명: "API_SPEC.md의 POST /api/tickets 명세를 확인하고 구현에 필요한 요구사항을 정리해줘."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 제목만으로 빠르게 등록 (우선순위: P1)

사용자가 제목만 입력해 새로운 할 일을 Backlog에 빠르게 등록할 수 있다.

**우선순위 이유**: 티켓 생성은 조회·수정·드래그앤드롭 등 보드의 다른 모든 기능의 전제 조건이며,
이 기능 없이는 앱을 사용할 수 없는 MVP 핵심 기능이다.

**독립 테스트**: 제목만 입력해 생성 요청을 보내고, Backlog 칼럼 맨 위에 새 티켓이 나타나는지
확인하는 것만으로 독립적으로 테스트 가능하다.

**인수 시나리오**:

1. **조건**: Backlog에 티켓이 없는 상태 / **행동**: 사용자가 제목만 입력해 생성을 요청 / **결과**:
   상태는 BACKLOG, 우선순위는 MEDIUM, position은 0인 티켓이 생성된다.
2. **조건**: Backlog에 기존 티켓들이 있는 상태 / **행동**: 사용자가 새 티켓을 생성 / **결과**: 새
   티켓이 기존 티켓들보다 앞선 순서 값으로 맨 위에 배치된다.

---

### 사용자 스토리 2 - 상세 정보를 포함해 등록 (우선순위: P2)

사용자가 설명, 우선순위, 계획시작일, 종료예정일을 함께 입력해 상세한 티켓을 등록할 수 있다.

**우선순위 이유**: 단순 할 일 목록을 넘어 일정과 우선순위 관리가 필요한 사용자를 지원하기 위함이며,
P1이 없으면 의미가 없는 확장 기능이다.

**독립 테스트**: 모든 선택 필드를 채워 생성 요청을 보낸 뒤, 응답에 입력한 값이 그대로 반영되는지
확인하는 것으로 독립적으로 테스트 가능하다.

**인수 시나리오**:

1. **조건**: 유효한 모든 필드 값 / **행동**: 생성을 요청 / **결과**: 입력한 값이 모두 포함된 티켓이
   생성된다.
2. **조건**: 종료예정일이 오늘 이전인 값 / **행동**: 생성을 요청 / **결과**: 생성이 거부되고 오류가
   반환된다.

---

### 사용자 스토리 3 - 잘못된 입력에 대한 명확한 안내 (우선순위: P3)

사용자가 유효하지 않은 값을 입력했을 때 무엇이 잘못됐는지 명확한 메시지로 안내받는다.

**우선순위 이유**: 사용자가 실패 원인을 알아야 재입력으로 작업을 완료할 수 있으며, 이는 앞선 두
스토리가 이미 동작하는 상태에서 사용성을 개선하는 계층이다.

**독립 테스트**: 제목 누락, 제목 초과, 설명 초과, 잘못된 우선순위, 과거 날짜 등 각 잘못된 입력
케이스별로 생성 요청을 보내고, 그에 대응하는 구체적 오류 메시지를 받는지 확인한다.

**인수 시나리오**:

1. **조건**: 제목 없이 요청 / **행동**: 생성을 요청 / **결과**: "제목을 입력해주세요" 오류를 받는다.
2. **조건**: 제목이 공백 문자로만 구성된 요청 / **행동**: 생성을 요청 / **결과**: "제목을
   입력해주세요" 오류를 받는다.
3. **조건**: 200자를 초과하는 제목 / **행동**: 생성을 요청 / **결과**: "제목은 200자 이내로
   입력해주세요" 오류를 받는다.
4. **조건**: 1000자를 초과하는 설명 / **행동**: 생성을 요청 / **결과**: "설명은 1000자 이내로
   입력해주세요" 오류를 받는다.
5. **조건**: 정의되지 않은 우선순위 값 / **행동**: 생성을 요청 / **결과**: "우선순위는 LOW, MEDIUM,
   HIGH 중 선택해주세요" 오류를 받는다.

---

### 예외 상황

- 설명(description)을 생략하면 값이 없는 상태로 저장되어야 한다.
- Backlog 칼럼이 완전히 비어 있는 상태에서 첫 티켓을 생성하면 맨 앞 순서 값은 0이어야 한다.
- 제목이 공백 문자로만 구성된 경우 제목 누락과 동일하게 취급한다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 사용자가 title(1~200자, 공백만으로 구성 불가)만으로 새 티켓을 생성할 수 있게
  해야 한다.
- **FR-002**: 시스템은 description(선택, 최대 1000자), priority(선택, LOW/MEDIUM/HIGH 중 하나, 기본값
  MEDIUM), plannedStartDate(선택), dueDate(선택, 오늘 이후 날짜) 입력을 지원해야 한다.
- **FR-003**: 시스템은 새로 생성되는 모든 티켓의 상태를 항상 BACKLOG로 시작해야 한다.
- **FR-004**: 시스템은 새 티켓을 Backlog 칼럼의 맨 위(가장 앞선 순서)에 배치해야 하며, 칼럼이 비어
  있으면 맨 앞 순서 값은 0이어야 한다.
- **FR-005**: 시스템은 다음 각 입력 오류에 대해 사용자가 원인을 알 수 있는 구체적 메시지를 반환해야
  한다: 제목 누락/공백, 제목 200자 초과, 설명 1000자 초과, 잘못된 priority 값, 오늘 이전의 dueDate.
- **FR-006**: 시스템은 위 검증 중 하나라도 실패하면 티켓을 생성하지 않고 요청 전체를 거부해야 한다.
- **FR-007**: 시스템은 생성에 성공하면 입력값과 자동 생성된 식별자·생성/수정 시각·파생 정보를 포함한
  생성된 티켓의 전체 정보를 사용자에게 돌려줘야 한다.

### 주요 엔터티

- **Ticket**: 칸반 보드의 작업 단위. title, description, status(BACKLOG/TODO/IN_PROGRESS/DONE),
  priority(LOW/MEDIUM/HIGH), position(칼럼 내 정렬 순서), plannedStartDate, dueDate, startedAt,
  completedAt, createdAt, updatedAt, isOverdue(파생 정보) 속성을 가진다. 생성 시점에는 status가
  BACKLOG로 고정되고 position은 칼럼 내 최상단 값으로 설정된다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 사용자는 제목 하나만 입력해 새 할 일을 보드에 추가할 수 있다.
- **SC-002**: 잘못된 입력이 발생한 모든 경우에서 사용자는 무엇이 문제인지 설명하는 구체적인 메시지를
  받는다.
- **SC-003**: 새로 생성된 티켓은 항상 Backlog 칼럼의 맨 위에 나타나 사용자가 추가 조작 없이 바로
  확인할 수 있다.
- **SC-004**: 설명·우선순위·일정 등 상세 필드를 포함해 생성해도 입력한 정보가 누락이나 변형 없이 그대로
  반영된다.

## 가정사항

- 별도의 인증/권한 체계 없이 단일 사용자(또는 신뢰된 사용자 그룹)가 보드를 사용한다고 가정한다 (제품/
  기술 요구사항에 인증 관련 요구사항이 없음).
- 동일한 제목을 가진 티켓의 중복 생성은 허용된다 (제목 고유성 제약 없음).
- 날짜 비교(예: 종료예정일이 "오늘 이후"인지 판단)는 서버 기준 날짜를 사용한다.
