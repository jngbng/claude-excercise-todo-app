# 기능 명세: 티켓 완료 처리

**기능 브랜치**: `006-complete-ticket`

**작성일**: 2026-07-24

**상태**: 초안

**입력**: 사용자 설명: "docs/API_SPECS.md의 PATCH /api/tickets/:id/complete(티켓 완료 처리,
FR-005)를 구현한다. DONE이 아닌 티켓을 호출하면 DONE으로 전환하고 completedAt을 기록하며,
이미 DONE인 티켓을 다시 호출하면 IN_PROGRESS로 되돌리고 completedAt을 지운다(토글). 존재하지
않는 ID는 404를 반환한다."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 티켓 완료 처리 및 되돌리기 (우선순위: P1)

사용자가 카드를 완료 처리하거나, 완료된 카드를 다시 진행 중으로 되돌릴 수 있다.

**우선순위 이유**: "완료"는 단순 칼럼 이동과 달리 완료 시각을 기록하고 24시간 노출 규칙과
맞물리는 별도의 의미를 가지므로, 조회·수정과는 독립적으로 다뤄야 하는 핵심 워크플로 기능이다.

**독립 테스트**: 완료 처리되지 않은 티켓과 이미 완료된 티켓 각각에 대해 같은 요청을 보내
토글 동작이 올바른지 확인한다.

**인수 시나리오**:

1. **조건**: 완료되지 않은 티켓 / **행동**: 완료 처리를 요청 / **결과**: 완료 상태가 되고
   완료 시각이 기록된다.
2. **조건**: 이미 완료된 티켓 / **행동**: 완료 처리를 다시 요청 / **결과**: 진행 중 상태로
   되돌아가고 완료 시각이 지워진다.
3. **조건**: 존재하지 않는 티켓 / **행동**: 완료 처리를 요청 / **결과**: 티켓을 찾을 수 없다는
   사실이 안내된다.

---

### 예외 상황

- 완료 처리는 상태와 관계없이 항상 토글로 동작한다 — 별도의 "이미 완료됨" 오류는 없다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-601**: 시스템은 완료되지 않은 티켓을 완료 상태로 전환하면서 완료 시각을 기록해야
  한다.
- **FR-602**: 시스템은 이미 완료된 티켓에 같은 조작을 반복하면 완료 이전 상태(진행 중)로
  되돌리고 완료 시각을 지워야 한다.
- **FR-603**: 시스템은 존재하지 않는 티켓에 대한 완료 처리 요청에 "찾을 수 없음"으로
  응답해야 한다.

### 주요 엔터티

- **Ticket**: `specs/001-create-ticket/spec.md`에서 정의한 것과 동일한 엔터티. 이 기능은
  status와 completedAt 필드의 토글 전환만 다룬다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-601**: 완료 처리와 되돌리기를 반복해도 완료 시각 기록이 매번 정확하게 갱신되거나
  지워진다.
- **SC-602**: 존재하지 않는 티켓의 완료 처리 시도는 항상 명확한 "찾을 수 없음" 안내를 받는다.

## Assumptions

- 인증/권한 체계가 없다는 가정은 `specs/001-create-ticket/spec.md`와 동일하게 유지된다.
- "완료 이전 상태로 되돌린다"는 항상 진행 중(IN_PROGRESS)으로 되돌리는 것을 의미한다 —
  `docs/API_SPECS.md`가 명시적으로 이렇게 정의한다.
