# 기능 명세: 드래그앤드롭으로 상태·순서 변경

**기능 브랜치**: `007-reorder-tickets`

**작성일**: 2026-07-24

**상태**: 초안

**입력**: 사용자 설명: "docs/API_SPECS.md의 PATCH /api/tickets/reorder(티켓 순서/상태 변경,
드래그앤드롭, FR-007)를 구현한다. 티켓을 BACKLOG/TODO/IN_PROGRESS 사이의 다른 칼럼으로
이동하거나 같은 칼럼 내 순서를 변경한다. DONE으로의 이동은 이 엔드포인트에서 허용하지
않는다(완료 처리 기능 전용). TODO 진입 시 startedAt을 기록하고 TODO->BACKLOG 복귀 시
초기화한다. 삽입 위치 주변 카드 간격이 좁으면 칼럼 전체를 재정렬한다. 트랜잭션으로 원자성을
보장한다."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 드래그앤드롭으로 상태·순서 변경 (우선순위: P1)

사용자가 카드를 다른 칼럼이나 같은 칼럼의 다른 위치로 옮기면, 그 결과가 저장되고 관련 날짜
정보(시작일 등)가 자동으로 갱신된다.

**우선순위 이유**: 칸반 보드의 핵심 가치는 상태 전환을 시각적으로 조작하는 것이다 — 조회만
되고 이동이 안 되면 정적인 목록과 다를 것이 없다.

**독립 테스트**: 티켓을 다른 칼럼으로 옮기는 요청과 같은 칼럼 내 순서만 바꾸는 요청을 각각
보내고, 결과 상태·순서·날짜 필드가 기대대로 바뀌는지 확인한다.

**인수 시나리오**:

1. **조건**: Backlog에 있는 티켓 / **행동**: To Do 칼럼으로 이동 / **결과**: 상태가 To Do로
   바뀌고 시작일이 현재 시각으로 기록된다.
2. **조건**: To Do에 있는 티켓 / **행동**: Backlog로 되돌림 / **결과**: 상태가 Backlog로
   바뀌고 시작일이 초기화된다.
3. **조건**: 같은 칼럼에 여러 티켓이 있는 상태 / **행동**: 그중 하나를 같은 칼럼 내 다른
   위치로 이동 / **결과**: 상태는 유지된 채 칼럼 내 순서만 바뀐다.
4. **조건**: 임의의 티켓 / **행동**: Done 칼럼으로의 이동을 시도 / **결과**: 요청이
   거부된다(Done 전환은 완료 처리 전용 기능으로만 가능하다).
5. **조건**: 존재하지 않는 티켓 / **행동**: 이동을 시도 / **결과**: 요청이 거부되고 티켓을
   찾을 수 없다는 사실이 안내된다.
6. **조건**: 칼럼 내 카드들 사이에 새 카드를 끼워 넣을 여유 공간이 없는 상태 / **행동**: 그
   사이로 이동을 시도 / **결과**: 이동은 성공하고, 해당 칼럼 전체의 순서가 다시 고르게
   재배치된다.

---

### 예외 상황

- 잘못된 상태 값(정의되지 않은 값 포함)으로의 이동 시도는 Done 이동 시도와 동일하게 거부된다.
- 재정렬은 이동 중인 카드를 포함한 칼럼 전체에 적용되어, 이동 결과와 재정렬 결과가 하나의
  트랜잭션으로 함께 반영된다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-701**: 시스템은 티켓을 Backlog/To Do/In Progress 사이의 다른 칼럼이나 같은 칼럼 내
  다른 위치로 이동할 수 있게 해야 하며, Done으로의 이동은 이 기능으로 허용하지 않아야 한다.
- **FR-702**: 시스템은 어떤 칼럼으로든 To Do로 이동하면 시작일을 기록하고, To Do에서
  Backlog로 이동하면 시작일을 지워야 한다.
- **FR-703**: 시스템은 그 외의 이동(예: Backlog↔In Progress, 같은 칼럼 내 순서 변경)에서는
  시작일을 변경하지 않아야 한다.
- **FR-704**: 시스템은 칼럼 내 카드 사이에 새 위치를 끼워 넣을 여유가 없을 때, 이동 자체는
  성공시키고 해당 칼럼 전체의 순서를 다시 고르게 재배치해야 한다.
- **FR-705**: 시스템은 존재하지 않는 티켓에 대한 이동 요청에 "찾을 수 없음"으로 응답해야
  한다.
- **FR-706**: 시스템은 이동과 재배치를 하나의 단위로 처리해, 일부만 반영된 채 중단되는 상태가
  없도록 해야 한다.

### 주요 엔터티

- **Ticket**: `specs/001-create-ticket/spec.md`에서 정의한 것과 동일한 엔터티. 이 기능은
  status(BACKLOG/TODO/IN_PROGRESS), position(칼럼 내 순서), startedAt 필드의 전환만
  다룬다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-701**: 사용자가 카드를 드래그해 다른 칼럼이나 위치로 옮기면, 그 결과가 다음 조회에서
  100% 정확히 반영되어 있다.
- **SC-702**: 사용자가 잘못된 이동(Done으로의 직접 이동, 존재하지 않는 티켓)을 시도하면
  항상 명확한 사유와 함께 거부된다.
- **SC-703**: 카드 사이 공간이 부족한 상황에서도 이동은 항상 성공하고, 재배치 이후에도 칼럼
  내 순서는 사용자가 의도한 대로 유지된다.

## Assumptions

- 순서 변경 요청의 위치 값은 클라이언트가 계산해 전달하는 대상 칼럼 내 삽입 지점(이동 중인
  티켓을 제외한 기존 카드 기준 0-based 인덱스)을 의미하며, 실제로 저장되는 순서 값 자체는
  서버가 기존 카드들과의 관계를 바탕으로 재계산한다(`docs/TRD.md` 드래그앤드롭 흐름 참고).
- 인증/권한 체계가 없다는 가정은 `specs/001-create-ticket/spec.md`와 동일하게 유지된다.
- Done 상태에 있는 티켓을 이 기능으로 이동시키려는 시도는 대상 상태 값 자체가 허용되지 않는
  값(DONE)이므로 거부되며, 별도의 "현재 상태가 DONE인 티켓" 특수 처리는 필요하지 않다.
