# 기능 명세: 티켓 단건 조회

**기능 브랜치**: `003-get-ticket-detail`

**작성일**: 2026-07-24

**상태**: 초안

**입력**: 사용자 설명: "docs/API_SPECS.md의 GET /api/tickets/:id(티켓 단건 조회, FR-003)를
구현한다. 존재하는 티켓은 전체 정보(isOverdue 포함)를 반환하고, 존재하지 않는 ID는 404
NOT_FOUND를 반환한다."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 티켓 상세 정보 확인 (우선순위: P1)

사용자가 보드에서 카드를 클릭하면 그 티켓의 모든 정보를 확인할 수 있다.

**우선순위 이유**: 보드에는 요약 정보만 보이므로, 상세 확인은 수정·완료·삭제 등 다른 조작을
하기 전에 필요한 조회 기능이며 이 기능 없이는 상세 화면 자체가 동작할 수 없다.

**독립 테스트**: 존재하는 티켓과 존재하지 않는 티켓 각각을 조회해 응답을 확인하는 것으로
독립적으로 테스트 가능하다.

**인수 시나리오**:

1. **조건**: 존재하는 티켓 / **행동**: 상세 조회 / **결과**: 해당 티켓의 전체 정보를 받는다.
2. **조건**: 존재하지 않는 티켓 / **행동**: 상세 조회 / **결과**: 티켓을 찾을 수 없다는 사실이
   안내된다.

---

### 예외 상황

- 완료된 지 24시간이 지나 보드 조회 결과에서는 빠진 Done 티켓도, 단건 조회로는 계속 접근할 수
  있어야 한다 (삭제되는 것이 아니므로).

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-301**: 시스템은 ID로 특정 티켓 하나의 전체 정보를 조회할 수 있게 해야 한다.
- **FR-302**: 시스템은 존재하지 않는 ID로 조회를 시도하면 그 사실을 명확히 안내해야 한다.

### 주요 엔터티

- **Ticket**: `specs/001-create-ticket/spec.md`에서 정의한 것과 동일한 엔터티. 이 기능은 읽기
  전용으로 전체 필드를 반환하며 값을 변경하지 않는다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-301**: 사용자는 존재하는 어떤 티켓이든 상세 정보를 100% 정확하게 확인할 수 있다.
- **SC-302**: 존재하지 않는 티켓을 조회하면 항상 명확한 "찾을 수 없음" 안내를 받는다.

## Assumptions

- 인증/권한 체계가 없다는 가정은 `specs/001-create-ticket/spec.md`와 동일하게 유지된다.
