# 기능 명세: 칸반 보드 전체 조회

**기능 브랜치**: `002-get-board`

**작성일**: 2026-07-24

**상태**: 초안

**입력**: 사용자 설명: "docs/API_SPECS.md의 GET /api/tickets(보드 전체 티켓 조회, 칼럼별 그룹화,
FR-002)를 구현한다. Backlog/To Do/In Progress/Done 4개 칼럼으로 그룹화하고 각 칼럼은 position
오름차순, Done 칼럼은 완료된 지 24시간 이내인 티켓만 표시한다."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 칸반 보드 한눈에 보기 (우선순위: P1)

사용자가 보드를 열면 모든 티켓이 상태별 칼럼(Backlog/To Do/In Progress/Done)으로 그룹화되어,
칼럼 내에서는 정해진 순서대로 나타난다.

**우선순위 이유**: 보드 조회는 다른 어떤 조작(수정·완료·삭제·이동)도 무엇에 대해 할지 알 수 있게
해주는 전제 조건이며, 이것 없이는 앱을 사용할 수 없는 핵심 기능이다.

**독립 테스트**: 티켓이 없는 상태와 여러 상태의 티켓이 섞여 있는 상태 각각에서 보드 조회를 호출해,
반환되는 그룹·정렬·필터링이 올바른지 확인하는 것만으로 독립적으로 테스트 가능하다.

**인수 시나리오**:

1. **조건**: 티켓이 하나도 없는 상태 / **행동**: 보드를 조회 / **결과**: 4개 칼럼 모두 빈 배열로
   반환된다.
2. **조건**: 여러 칼럼에 걸쳐 티켓이 존재 / **행동**: 보드를 조회 / **결과**: 각 티켓이 자신의
   상태에 해당하는 칼럼에 나타나고, 같은 칼럼 내에서는 정해진 순서대로 정렬되어 있다.
3. **조건**: Done으로 완료된 지 24시간이 지난 티켓이 존재 / **행동**: 보드를 조회 / **결과**: 그
   티켓은 Done 칼럼 결과에 나타나지 않는다.

---

### 예외 상황

- Done 칼럼에서 24시간 경과로 제외된 티켓도 삭제된 것은 아니므로, 이후 단건 조회 기능(다른
  기능에서 다룸)으로는 계속 접근 가능해야 한다 — 이 기능은 그 티켓을 보드 조회 결과에서만
  제외한다.
- 어떤 칼럼에도 티켓이 없을 수 있으며, 이 경우 해당 칼럼은 빈 배열로 반환된다(칼럼 자체가
  생략되지 않는다).

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-201**: 시스템은 전체 티켓을 상태별 4개 그룹(Backlog/To Do/In Progress/Done)으로 나누어
  조회할 수 있게 해야 하며, 4개 그룹은 티켓이 없어도 항상 빈 배열로 함께 반환되어야 한다.
- **FR-202**: 시스템은 각 그룹 내에서 티켓을 정해진 순서(생성/이동 시 결정된 순서)대로 정렬해
  반환해야 한다.
- **FR-203**: 시스템은 완료된 지 24시간이 지난 티켓을 Done 그룹 결과에서 제외해야 한다.

### 주요 엔터티

- **Ticket**: `specs/001-create-ticket/spec.md`에서 정의한 것과 동일한 엔터티. 이 기능은 그
  status(BACKLOG/TODO/IN_PROGRESS/DONE), position(칼럼 내 순서), completedAt 필드를 읽어
  그룹화·정렬·필터링만 수행하며 값을 변경하지 않는다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-201**: 사용자는 보드를 한 번 조회하는 것만으로 모든 티켓의 현재 상태와 순서를 정확히
  파악할 수 있다.
- **SC-202**: 완료된 지 24시간이 넘은 티켓은 100% Done 목록에서 사라져, 사용자가 오래된 완료
  항목 때문에 화면이 어수선해지는 경험을 하지 않는다.

## Assumptions

- 인증/권한 체계가 없다는 가정은 `specs/001-create-ticket/spec.md`와 동일하게 유지된다.
- "정해진 순서"의 실제 값(position)이 어떻게 만들어지는지는 티켓 생성·이동 기능의 책임이며, 이
  기능은 이미 저장된 순서 값을 그대로 읽어 정렬만 한다.
