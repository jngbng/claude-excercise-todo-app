# 기능 명세: 티켓 수정

**기능 브랜치**: `004-update-ticket`

**작성일**: 2026-07-24

**상태**: 초안

**입력**: 사용자 설명: "docs/API_SPECS.md의 PATCH /api/tickets/:id(티켓 수정, FR-004)를
구현한다. 전송된 필드만 부분 수정하고(title/description/priority/plannedStartDate/dueDate),
description/plannedStartDate/dueDate는 null 전송 시 삭제한다. 생성 시와 동일한 유효성 규칙을
적용하고, 존재하지 않는 ID는 404를 반환한다."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 티켓 내용 수정 (우선순위: P1)

사용자가 상세 화면에서 제목·설명·우선순위·일정을 고쳐 저장할 수 있다.

**우선순위 이유**: 생성 시 입력을 잘못했거나 계획이 바뀌었을 때 티켓을 다시 만들지 않고 고칠
수 있어야 하며, 이는 티켓 관리 기능의 핵심 가치다.

**독립 테스트**: 필드 일부만 보내는 요청과 여러 필드를 함께 보내는 요청 각각에 대해, 보낸
필드만 바뀌고 나머지는 그대로인지 확인한다.

**인수 시나리오**:

1. **조건**: 존재하는 티켓 / **행동**: 제목만 바꿔 저장 / **결과**: 제목만 바뀌고 나머지
   필드는 그대로 유지된다.
2. **조건**: 설명이 있는 티켓 / **행동**: 설명을 비우도록 저장 / **결과**: 설명이 삭제된다.
3. **조건**: 존재하는 티켓 / **행동**: 제목·우선순위·종료예정일을 한 번에 저장 / **결과**:
   보낸 필드가 모두 반영된다.
4. **조건**: 존재하지 않는 티켓 / **행동**: 수정을 시도 / **결과**: 티켓을 찾을 수 없다는
   사실이 안내된다.
5. **조건**: 200자를 초과하는 제목 또는 오늘 이전의 종료예정일 / **행동**: 수정을 시도 /
   **결과**: 요청이 거부되고 원인이 안내된다.

---

### 예외 상황

- 아무 필드도 보내지 않고 수정을 요청하면 아무것도 바뀌지 않은 채 그대로의 티켓 정보가
  반환된다.
- 계획시작일·종료예정일도 설명과 동일하게 `null` 전송 시 삭제할 수 있다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-401**: 시스템은 티켓의 제목·설명·우선순위·계획시작일·계획종료일 중 전송된 필드만
  선택적으로 수정할 수 있게 해야 하며, 나머지 필드는 그대로 유지해야 한다.
- **FR-402**: 시스템은 설명·계획시작일·계획종료일에 대해 명시적으로 `null`을 전송하면 해당
  필드를 지울 수 있게 해야 한다.
- **FR-403**: 시스템은 수정 시에도 생성 시와 동일한 제목·설명·우선순위·날짜 유효성 규칙을
  적용해야 한다.
- **FR-404**: 시스템은 존재하지 않는 티켓에 대한 수정 요청에 "찾을 수 없음"으로 응답해야
  한다.

### 주요 엔터티

- **Ticket**: `specs/001-create-ticket/spec.md`에서 정의한 것과 동일한 엔터티. 이 기능은
  title/description/priority/plannedStartDate/dueDate 필드의 부분 수정만 다룬다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-401**: 사용자는 티켓 일부 필드만 수정해도 나머지 정보를 다시 입력할 필요가 없다.
- **SC-402**: 잘못된 입력으로 수정을 시도하면 항상 명확한 사유와 함께 거부되고, 기존 값은
  그대로 보존된다.

## Assumptions

- 인증/권한 체계가 없다는 가정은 `specs/001-create-ticket/spec.md`와 동일하게 유지된다.
- status/position/startedAt/completedAt은 이 기능의 수정 대상이 아니다(다른 기능의 책임).
